# Implementation Plan: Quick Match (Partida Rápida)

**Branch**: `002-quick-match` | **Date**: 2026-05-19 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/002-quick-match/spec.md`

## Summary

Add a Quick Match button that enqueues the authenticated player for a given `gamesToPlay` value and
automatically pairs them with the first compatible opponent in the queue. When paired, a private
match is created and started immediately; both players are notified via the existing
`/user/queue/match` WebSocket channel. A player in the queue is blocked from all other match actions
(FR-010), and the queue is cleaned up automatically on disconnect (FR-007).

The technical approach uses an ephemeral in-memory queue (no persistence), a new
`Match.quickMatch()`
domain factory, and an extension to `PlayerAvailabilityChecker` — all following the existing
Hexagonal/DDD patterns of the codebase.

---

## Technical Context

**Language/Version**: Java 21, Spring Boot 3.x (existing project)

**Primary Dependencies**: Spring WebSocket/STOMP (existing), Spring Data JPA (existing),
ConcurrentLinkedDeque (JDK built-in)

**Storage**: In-memory only (`ConcurrentHashMap` + `ConcurrentLinkedDeque`) — ephemeral by design.
No new DB table, no Flyway migration needed.

**Testing**: JUnit 5 + Mockito (existing); H2 in-memory for integration tests; architecture tests
via ArchUnit (existing `CleanArchitectureTest`)

**Target Platform**: Linux server (existing)

**Performance Goals**: Pairing notification delivered within 2 seconds of second player joining
queue (SC-001). Queue operations O(1) amortised.

**Constraints**: 70% minimum JaCoCo line coverage; ArchUnit layering rules enforced at build time;
no Spring imports in domain or application layers.

**Scale/Scope**: Single-server, in-memory queue — suitable for current scale. Horizontal scaling
would require a distributed queue (future concern, out of scope).

## Implementation Layers — Detailed Design

### Domain Layer

#### `QuickMatchTicket`

Immutable class (not `AggregateBase` — no state machine). Fields:

- `PlayerId playerId`
- `GamesToPlay gamesToPlay`
- `Instant enqueuedAt`
- `String webSocketSessionId` (for disconnect cleanup; may be null if unavailable)

Constructor validates non-null `playerId` and `gamesToPlay`.

#### `PlayerAlreadyInQueueException`

Follows the same base class as `PlayerAlreadyInActiveMatchException` — consistent domain exception.

#### `QuickMatchQueuePort`

Domain output port in `com.villo.truco.domain.ports`:

```java
void enqueue(QuickMatchTicket ticket);

Optional<QuickMatchTicket> tryDequeue(PlayerId playerId);

Optional<QuickMatchTicket> tryMatchOpponent(PlayerId enqueuingPlayer, GamesToPlay gamesToPlay);

boolean isPlayerQueued(PlayerId playerId);

Optional<QuickMatchTicket> tryDequeueBySessionId(String sessionId);
```

#### `Match.quickMatch()` — new static factory

Creates a match with both players assigned, `Visibility.PRIVATE`, transitions directly to
`IN_PROGRESS` by calling `startInternal()`. Does NOT publish `PublicMatchLobbyOpenedEvent`.
DOES publish `MatchStartedEvent` (or equivalent) so the existing notification pipeline delivers
the match ID to both players on `/user/queue/match`.

---

### Application Layer

#### `EnqueueForQuickMatchCommandHandler` — orchestration

```
1. if queuePort.isPlayerQueued(playerId) → return current ticket as SEARCHING (idempotency, FR-008)
2. playerAvailabilityChecker.ensureAvailable(playerId)  ← throws if unavailable
3. opponent = queuePort.tryMatchOpponent(playerId, gamesToPlay)
4. if opponent present:
     a. match = Match.quickMatch(opponent.playerId(), playerId, gamesToPlay)
     b. matchRepository.save(match)
     c. matchEventNotifier.publishDomainEvents(match.getDomainEvents())
     d. return QuickMatchSearchDTO(MATCHED, match.getId(), now)
5. else:
     a. ticket = new QuickMatchTicket(playerId, gamesToPlay, now, sessionId)
     b. queuePort.enqueue(ticket)
     c. return QuickMatchSearchDTO(SEARCHING, null, now)
```

#### `CancelQuickMatchSearchCommandHandler`

```
1. queuePort.tryDequeue(playerId)   ← no-op if not in queue
2. return void
```

#### `PlayerAvailabilityChecker` — change

Add `QuickMatchQueuePort` constructor parameter. After the `rematchSessionRepository` check:

```java
if(this.quickMatchQueuePort.isPlayerQueued(playerId)){
    throw new

PlayerAlreadyInQueueException();
}
```

### Infrastructure Layer

#### `InMemoryQuickMatchQueueAdapter`

`@Component` implementing `QuickMatchQueuePort`:

```java
ConcurrentHashMap<GamesToPlay, ConcurrentLinkedDeque<QuickMatchTicket>> queues;
ConcurrentHashMap<PlayerId, QuickMatchTicket> byPlayer;    // O(1) lookup
ConcurrentHashMap<String, PlayerId> bySessionId;           // O(1) disconnect lookup
```

`tryMatchOpponent()` synchronized on the per-gamesToPlay deque to atomically find-and-remove.

#### `QuickMatchSessionDisconnectEventListener`

`@Component`, `@EventListener(SessionDisconnectEvent.class)`. Extracts `PlayerId` from session
principal via `StompHeaderAccessor.wrap(event.getMessage()).getUser()`. Calls
`cancelQuickMatchSearch.handle(new CancelQuickMatchSearchCommand(playerId))`. Wrapped in try-catch
to avoid breaking the disconnect lifecycle.

#### `QuickMatchController`

```java
@RestController @RequestMapping("/api/matches/quick")
```

- `POST /` → `EnqueueForQuickMatchUseCase`
- `DELETE /` → `CancelQuickMatchSearchUseCase`

Follows identical pattern to `MatchController`: extract `Jwt` principal, delegate to use case.

#### `QuickMatchUseCaseConfiguration`

`@Configuration` — same pattern as `MatchUseCaseConfiguration`. Wraps handlers with
`retryTransactionalPipeline` and exposes as use case interfaces.

#### `PlayerAvailabilityConfiguration` — change

Add `QuickMatchQueuePort` parameter to the `@Bean` factory and pass it to
`PlayerAvailabilityChecker`.

## Open Questions (resolve during implementation)

1. **WebSocket session ID in HTTP context**: Can `QuickMatchController` (HTTP) access the player's
   WebSocket session ID? If not, disconnect cleanup relies on `PlayerId` from the STOMP principal
   alone — `bySessionId` map is unused. Verify during implementation; simplify if unavailable.

2. **`Match.quickMatch()` event details**: Verify which domain events `startInternal()` currently
   publishes and confirm the existing WebSocket notification pipeline delivers match details
   (both player IDs, gamesToPlay) to both `/user/queue/match` subscribers.

3. **`matchEventNotifier` for quick match**: Confirm the existing `MatchEventNotifier` bean (used
   in `CreateMatchCommandHandler`) can be reused in `EnqueueForQuickMatchCommandHandler` without
   a separate notifier.
